# Goal

Fix `wt` hanging silently and leaving terminal escape sequences in the prompt
after Ctrl+C. User report:

```
wt
^C
~/HomeProjects/worktree-cli  on   master  10;rgb:ffff/ffff/ffff   ✔ ...
~/HomeProjects/worktree-cli  on   master  11;rgb:2828/2c2c/3434   1 ✘ ...
~/HomeProjects/worktree-cli  on   master  wt>|ghostty 1.3.1R...    1 ✘ ...
```

Those are OSC 10/11 (fg/bg color queries), DA2 (`>|ghostty 1.3.1`),
mode queries — all generated by OpenTUI's `setupTerminal()` and leaking
into the captured stdout of the `wt()` shell wrapper.

# Why this happens

The shell wrapper installed by `install.sh` does:

```bash
out="$(command wt-bin "$@")"
```

`$()` replaces fd 1 with a pipe. OpenTUI writes its terminal capability
queries to fd 1 (via the Zig native module — `process.stdout` swap
doesn't help because Zig has its own fd 1). Those queries land in the
shell's `$out` instead of the terminal, and when the shell later
displays `$out`, the terminal interprets them as garbage.

# Approach chosen

Side-channel for the result path: add `--output-file <path>` to the
binary; have the shell wrapper pass a `mktemp` file. With the path going
to a file, the binary's stdout stays connected to the real terminal
(no `$()` capture), so all OpenTUI escape sequences go where they belong
and never pollute anything.

# Issues — open

## Issue 1 — `wt-bin` doesn't forward args (likely root cause of original bug)

`bin/wt-bin:3`:
```bash
exec bun run "$(dirname "$SCRIPT")/../src/index.tsx" "$@"
```

`bun run script.tsx <args>` does NOT pass `<args>` to the script —
`bun run` consumes them. Confirmed by reproducing in a temp repo: with
the `wt-bin` wrapper, `process.argv.slice(2)` is empty even when called
with a branch name.

Result: `wt feat/foo` always falls through to TUI mode (because
`args[0]` is undefined), regardless of any `--output-file` work.

**Fix:** change line 3 to one of:
- `exec bun "$(dirname "$SCRIPT")/../src/index.tsx" "$@"` (no `run`)
- `exec bun run "$(dirname "$SCRIPT")/../src/index.tsx" -- "$@"`

Without this fix, my `--output-file` path is unreachable from the
shell wrapper.

This also probably explains the *original* user complaint: the
intent was `wt` (no args, TUI), but with `bun run` swallowing args,
even `wt feat/foo` would fall through to TUI mode.

## Issue 2 — my `--output-file` arg-parsing has off-by-one bug

In `src/index.tsx`:

```ts
const outputFileIdx = args.indexOf("--output-file");      // -1 when absent
const positionalArgs = args.filter(
  (_, i) => i !== outputFileIdx && i !== outputFileIdx + 1,
);
// when outputFileIdx === -1: filter excludes index -1 (none) AND
// index 0 (the actual first positional arg!) → drops args[0].
```

This is what's making the tests fail — even after fixing Issue 1, the
non-interactive branch never sees the branch name because my filter
drops it.

**Fix:** guard the filter so `outputFileIdx + 1` only excludes when
`outputFileIdx !== -1`.
